## One-line status

The portal now has a **working, branded, config-driven editor — including featured-image upload — on top of the secure per-tenant bridge, and (as of B.08) it is keyboard-operable, screen-reader-legible, mobile-clean, and WCAG 2.1 AA on contrast, with that accessibility locked in by offline `axe` tests**: a signed-in user resolves server-side to the **one** client they own, and can **create, edit, save-as-draft, publish, delete, and attach a featured image to** a blog post (headline, body, summary, slug, image) on **that client's own Sanity project** — every write AND every image upload built from the session-resolved tenant's project + **decrypted** token and nothing else. The image is uploaded server-side through the same per-tenant client (never a browser token), validated (4 MB cap + JPG/PNG/WebP/GIF allowlist), and written onto the post as an asset reference (tri-state: preserve / write / clear); an existing image is surfaced in the edit form. Draft/published follows **Sanity's draft-id model** (save → `drafts.<id>`; publish → atomic promote-and-delete transaction; delete → both variants), consistent with the B.04 read reduction. The load-bearing **cross-tenant isolation test now also covers the image upload + image write** (the §5 gate): user A's create/save/publish/delete **and image upload** are dispatched only through A's project + token, and an unmapped/unauthenticated session is denied before any writer/uploader is built or any secret is read. Tokens stay **server-only** end to end (no `'use client'` module imports a token/resolver/mutation/Sanity module; the build's `server-only` guard holds; verified). **Live uploads/writes against a real Sanity project are M.02** — the B.03 placeholder project surfaces the friendly error state by design; the offline upload/write-isolation + unit tests are the authoritative B.06 proof. **B.07** settled how a publish reaches the live site: the refresh is **driven by each client's own Sanity → site webhook**, so the portal **sends nothing on publish and holds no revalidation secret** (`revalidateUrl` is retained-but-unused) — a docs-and-decision phase that adds no portal feature code and keeps the suite green; the per-client webhook setup is **M.01** and the live end-to-end proof is **M.02**.

**New in B.06 (featured-image upload — through the same locked path):**

## What is NOT built yet (by design)

- **No portal-side live-site revalidation call (by design, settled in B.07).** The portal deliberately does **not** call any client's `revalidate_url` and holds no revalidation secret; the live-site refresh is delegated to each client's own Sanity → site webhook (see "New in B.07" above and the 2026-06-28 Decisions entry). What is **not done yet** is the per-client *enablement*: configuring each client's Sanity publish webhook (manage.sanity.io) is **M.01**, and the live publish → live-site round-trip proof is **M.02**. Until a client's webhook is switched on, a portal publish still succeeds but that client's live site won't refresh.
- **No image cropping / hotspot / focal-point UI, alt-text editing, transformations, or per-locale / multiple images.** v1 is one featured image, non-localized (an unchanged image's existing hotspot/crop/alt metadata is preserved, never edited). Larger-than-4 MB / browser-direct uploads are explicitly deferred (see the 2026-06-27 Decisions entry).
- **No real client, real Sanity project, or real token.** B.06 still uses only the B.03 throwaway test client (placeholder project + dummy token), so a *live* read, write, or image upload errors by design (the friendly error/read-error state). Real Editor tokens are onboarded in **M.01**; the live end-to-end create → upload image → publish → delete per real client is **M.02**.
- **No new environment variables and no new dependencies.** The Sanity project id + token come from the registry, never from env. (B.06 *did* make the first `next.config.ts` change since B.01 — image `remotePatterns` + the Server Action body-size limit — but added no deps and no env vars.)
- **No full-fidelity Portable Text editing.** The editor edits the body as plain paragraphs; a body richer than simple paragraphs (marks, links, headings, lists, embeds) is shown **read-only** with an "edit in Sanity" note so it is never silently stripped. Full rich-body editing is out of scope.
- **No `status`-field writes.** Publish state is the draft-id model only; the schema `status` field is left in config but never written (per-client `status` handling, if needed, is M.01).
- **No multi-client operators, no password reset / key rotation / rate limiting, no public signup** — as before.

## Security posture (B.07 — write + image-upload path; publish refresh delegated to Sanity)

- **The portal sends nothing on publish and holds no revalidation secret (B.07).** Live-site refresh is delegated to each client's own Sanity → site webhook, so the portal makes **no outbound HTTP call to any client endpoint** and stores **no `SANITY_REVALIDATE_SECRET`** — that secret lives only on each client site (and in that client's Sanity webhook config), never in the portal or this repo. This **shrinks** the secret surface of the one app that holds editing keys to every client's blog (one fewer secret per client). The §5 write/upload isolation gate is unaffected because no new outbound call was added. See the 2026-06-28 Decisions entry and `docs/runbooks/live-site-revalidation.md`.
- **The decrypted token is a server-only secret, end to end.** It exists only inside `resolve-tenant.ts` / `sanity/*.ts` on the server. Every `lib/registry` and `lib/sanity` module — including the new `sanity/assets.ts` — starts with `import 'server-only'`; `field-map.ts` holds no secret and imports `TenantConfig` via `import type`. **Verified:** no `'use client'` file imports any token/resolver/mutation/Sanity module (the editor reaches the upload only through the `'use server'` `uploadImageAction`); `assets.ts` is imported only by `actions.ts` + the test; `npm run build` keeps the genuine `server-only` guard in force.
- **Image upload goes through the same locked per-tenant path — never a browser token.** `uploadImageAction` re-resolves + re-authorizes the tenant on every POST and uploads via the server-side per-tenant client built from the session owner's project + token; the bytes land in that one client's dataset and only the asset reference is written. A browser-direct upload (which would need a browser-exposed token) is explicitly out of scope (2026-06-27 Decisions entry). The upload is size- + type-validated server-side; a submitted asset id is shape-validated before it is trusted as a `_ref`; the token/project id/raw error never reach the `ImageUploadState` returned to the browser. The isolation test's §5 gate now covers the upload + the image write.
- **Authorize on every access AND every mutation, from the session.** The user's `client_id` is derived from the validated `getClaims()` session via the RLS-scoped client; **a caller may never supply a project id, client id, or token.** Every Server Action **re-resolves the tenant itself** (Server Actions are independently reachable by direct POST; a page/layout check does not protect them), so authorization is re-checked on every write. There is exactly one client a request may touch — the session owner's. Proven by the offline isolation test, now extended to all four mutations.
- **Caller-supplied post ids are sanitized, never trusted as selectors.** A submitted id is normalized (`drafts.`/`versions.` prefixes stripped; non-plain ids rejected) and only ever applied through the owner's per-tenant client — so an attacker-shaped id can at most touch a doc in the attacker's own dataset, never another tenant's project.
- **Defense in depth:** config reads go through the RLS-scoped client; the encrypted token is read only by the service-role client, keyed on the already-resolved `client_id`; the token is decrypted only after the user-scoped lookups succeed, and a tamper/wrong-key error propagates (never a plaintext fallback).
- **Fail closed, quietly.** Every resolver failure throws a typed `TenantResolutionError`; pages and actions turn those into friendly/generic states. The raw error, project id, token, and doc type are never rendered or returned.
- **GROQ injection guard.** `$type`/`$id`/`$draftId` are always bound parameters; all interpolated field names — read projections (incl. the B.06 image projection) AND mutation object keys (incl. the slug container and the image field) — are validated by `assertSafeFieldPath` / `assertWritableFieldPaths` before use.
- **Data-loss guard.** Save does a read-modify-write overlay (non-essential client fields preserved); a body too rich for plain-text editing is detected and shown read-only so it is never silently stripped; an **unchanged image is preserved untouched** (keeping any hotspot/crop/alt metadata) and is dropped only on an explicit clear — never silently overwritten.
- **No secret values anywhere in the diff** (code, comments, tests, report) — grep-verified; no `'use client'` file imports a token/resolver/mutation/Sanity module — the build's `server-only` guard holds. The isolation/mutation/upload tests use only invented, meaningless strings.

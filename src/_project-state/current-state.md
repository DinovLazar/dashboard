# Dashboard — Current State

> The full project snapshot. **Read this first every session.** Factual, not aspirational — it records what IS.
> Last updated: **2026-06-27** (end of Phase **B.05**).

## One-line status

The portal now has a **working, branded, config-driven editor on top of the secure per-tenant bridge**: a signed-in user resolves server-side to the **one** client they own, and can **create, edit, save-as-draft, publish, and delete** a blog post (headline, body, summary, slug; image field present but inert until B.06) on **that client's own Sanity project** — every write built from the session-resolved tenant's project + **decrypted** token and nothing else. Draft/published follows **Sanity's draft-id model** (save → `drafts.<id>`; publish → atomic promote-and-delete transaction; delete → both variants), consistent with the B.04 read reduction. The load-bearing **cross-tenant isolation test is now extended to writes** (the §5 gate): user A's create/save/publish/delete are dispatched only through A's project + token, and an unmapped/unauthenticated session is denied before any writer is built or any secret is read. Tokens stay **server-only** end to end (no `'use client'` module imports a token/resolver/mutation/Sanity module; the build's `server-only` guard holds; verified). **Live writes against a real Sanity project are M.02** — the B.03 placeholder project surfaces the friendly error state by design; the offline write-isolation + mutation unit tests are the authoritative B.05 proof.

**From B.01 (foundation + look):** public protected repo to the Vertex standard; Next 16 + React 19 + TS + Tailwind v4 + shadcn (`base-nova`) app that builds clean and deploys to Vercel; Vertex brand applied (dark-only); `src/_project-state/` as the docs home.

**From B.02 (Supabase auth — the locked door):** server-side Supabase auth (`@supabase/ssr` + `@supabase/supabase-js`); `src/proxy.ts` session refresh + redirect; functional `/login` (email + password Server Action, no signup, generic error); `(portal)` gated by a `getClaims()` server check; sign-out; no-cache hardening.

**From B.03 (registry data model — the secure address book):** three RLS-protected Supabase tables (`clients`, `client_users`, `client_secrets`) so a signed-in user can read only their own client + config and **cannot read any token**; app-layer **AES-256-GCM** token encryption (`src/lib/crypto/tokens.ts`); the service-role client (`src/lib/supabase/admin.ts`) as the only reader of `client_secrets`; seed/verify scripts + an operator runbook.

**New in B.04 (secure per-tenant Sanity bridge — read path):**
- **The tenant resolver** (`src/lib/registry/resolve-tenant.ts`, `server-only`): turns the logged-in user → `{ config, token }` for their one client. Derives the user id from the validated `getClaims()` session (`sub`); reads the user→client mapping and the client config through the **RLS-scoped** Supabase client; reads the encrypted token **only** through the **service-role** client, keyed on the already-resolved, user-owned `client_id`; decrypts with `decryptToken`. Fails closed with a typed `TenantResolutionError` (`unauthenticated` / `no-client` / `config-missing` / `secret-missing`). **No caller can supply a client/project id or token.** The pure core `resolveTenantWith(deps)` takes injectable seams; the production `resolveTenant` is `cache()`-deduplicated per request. Types in `src/lib/registry/types.ts` (`TenantConfig`, `TenantContext`).
- **The per-tenant Sanity client factory** (`src/lib/sanity/client.ts`, `server-only`): `createTenantSanityClient(config, token)` builds a fresh `@sanity/client` per request — `projectId`/`dataset`/`token`, `apiVersion: SANITY_API_VERSION` (`'2026-03-01'`), `useCdn: false`, `perspective: 'raw'` (so the editor sees drafts + published side by side). Throws on an empty token.
- **The read path** (`src/lib/sanity/posts.ts`, `server-only`): `listPosts(tenant, makeClient?)` fetches with the field-map query, reduces the `raw`-perspective variants to one `PostSummary` per logical post (status ∈ {draft, published}, `hasUnpublishedEdits`, draft-preferred display values, `versions.*` ignored), sorted newest-first. `makeClient` is injectable for the offline test. `displayValue` handles plain-string now / localized-object later. The token is never returned or logged.
- **The field-map → GROQ helper** (`src/lib/config/field-map.ts`): `assertSafeFieldPath` (regex guard against GROQ injection from a malformed registry row) + `buildPostListQuery` (binds `$type` as a **parameter**, interpolates only validated field names).
- **The wired `/posts` page** (`src/app/(portal)/posts/page.tsx`, Server Component): resolves the tenant, lists posts, and renders the client label + a populated list, or the empty / not-linked / not-ready / read-error states — never leaking the project id, token, or a raw error. Both "New post" buttons stay disabled (editor = B.05). The presentational list is `src/components/portal/posts-list.tsx` (gets only `PostSummary[]`, never the tenant/token).
- **Top-bar label** (`src/app/(portal)/layout.tsx`): now the resolved client label (best-effort; falls back to the email if resolution fails), sharing the page's `cache()`d resolution — no extra query.
- **The automated cross-tenant isolation test** (`src/lib/registry/isolation.test.ts`, offline): **read path** — A→A / B→B; the Sanity client is built with the session owner's projectId + token (never the other tenant's); no caller can override the client/project; the secret read is keyed to the owner's client id; the read path is unreachable for an unmapped (`no-client`) / null (`unauthenticated`) session — plus `config-missing` / `secret-missing` / decrypt-failure fail-closed paths. **Write path (B.05, the §5 gate)** — a recording writer seam proves every `createDraft`/`saveDraft`/`publishPost`/`deletePost` is BUILT with and DISPATCHED through the owner's project + token (never the other tenant's); an attacker-controlled/foreign post id still resolves only within the owner's dataset; an unmapped/null session is denied before any writer is constructed or any secret read; and the write functions expose no client/project/token selector (session identity is the sole input). Complemented by `posts.test.ts` (read reduce + `getPost`), `mutations.test.ts` (dispatch shapes), `field-map.test.ts` (injection guard + write-key validation), `localize.test.ts`, `portable-text.test.ts`, `doc-id.test.ts`. **105 tests pass.**

**New in B.05 (the config-driven editor — write path):**
- **The mutation module** (`src/lib/sanity/mutations.ts`, `server-only`): the **only** write site. `createDraft` / `saveDraft` / `publishPost` / `deletePost`, each taking a session-resolved `TenantContext` + an injectable writer factory (default = the real per-tenant client). Draft-id model: save = `createOrReplace('drafts.'+id)`; publish = one transaction (`createOrReplace` published from the draft's full content + `delete` the draft); delete = one transaction (both variants). Save does a **read-modify-write overlay** so the client's non-essential fields (image, author, categories, FAQs) are preserved. The token is never returned or logged.
- **The mutating Server Actions** (`src/app/(portal)/posts/actions.ts`, `'use server'`): `createPostAction` / `saveDraftAction` / `publishPostAction` / `deletePostAction`. Each **re-resolves the tenant itself** (Server Actions are reachable by direct POST), parses per-locale fields from `FormData`, dispatches the matching mutation, `revalidatePath`s, and returns a generic non-leaking result or `redirect`s. No tenant is passed from the page; no caller-supplied id/project/token is trusted (ids normalized).
- **The single-post read** (`src/lib/sanity/posts.ts` → `getPost` + `PostDetail`): loads one post for the edit form (draft-preferred reduce, per-locale field values, `bodyEditable` flag), id normalized; token stays server-side.
- **Pure helpers** (`src/lib/config/localize.ts`, `portable-text.ts`; `src/lib/sanity/doc-id.ts`, `src/lib/config/field-map.ts` extensions): localized value ⇄ per-locale form state; plain text ⇄ minimal Portable Text with a rich-body data-loss guard; id sanitization; the single-post query + write-key/slug-container validation.
- **The editor UI** (`src/components/editor/post-editor.tsx`, `'use client'`): branded, mobile-first form — per-locale headline/summary/body + slug, **locale tabs when multi-locale**, an inert "Image upload arrives soon" field, Save-draft / Publish, and Delete with a confirm step. Receives only non-secret serializable props (locales + per-locale values + status) — never the tenant/token. Plus the `/posts/new` and `/posts/[id]` pages and the shared `editor-message.tsx` friendly states; the `/posts` "New post" buttons now link to `/posts/new` and each list row links to `/posts/[id]`.

## What is NOT built yet (by design)

- **No image upload / no `next.config.ts` image `remotePatterns`** — the editor's image field is present but inert (disabled, "coming soon"); no image is written. **B.06**. **No publish → live-site revalidation** (the portal does not yet call the client site's `revalidate_url`) — **B.07**.
- **No real client, real Sanity project, or real token.** B.05 still uses only the B.03 throwaway test client (placeholder project + dummy token), so a *live* read or write errors by design (the friendly error/read-error state). Real Editor tokens are onboarded in **M.01**; the live end-to-end create → publish → delete per real client is **M.02**.
- **No new environment variables, no new dependencies, no `next.config.ts` change.** The Sanity project id + token come from the registry, never from env.
- **No full-fidelity Portable Text editing.** The editor edits the body as plain paragraphs; a body richer than simple paragraphs (marks, links, headings, lists, embeds) is shown **read-only** with an "edit in Sanity" note so it is never silently stripped. Full rich-body editing is out of scope.
- **No `status`-field writes.** Publish state is the draft-id model only; the schema `status` field is left in config but never written (per-client `status` handling, if needed, is M.01).
- **No multi-client operators, no password reset / key rotation / rate limiting, no public signup** — as before.

## Security posture (B.05 — write path)

- **The decrypted token is a server-only secret, end to end.** It exists only inside `resolve-tenant.ts` / `sanity/*.ts` on the server. Every new `lib/registry` and `lib/sanity` module starts with `import 'server-only'`; `field-map.ts` holds no secret and imports `TenantConfig` via `import type`. **Verified:** no `'use client'` file imports any of the token/resolver/Sanity modules; `npm run build` keeps the genuine `server-only` guard in force.
- **Authorize on every access AND every mutation, from the session.** The user's `client_id` is derived from the validated `getClaims()` session via the RLS-scoped client; **a caller may never supply a project id, client id, or token.** Every Server Action **re-resolves the tenant itself** (Server Actions are independently reachable by direct POST; a page/layout check does not protect them), so authorization is re-checked on every write. There is exactly one client a request may touch — the session owner's. Proven by the offline isolation test, now extended to all four mutations.
- **Caller-supplied post ids are sanitized, never trusted as selectors.** A submitted id is normalized (`drafts.`/`versions.` prefixes stripped; non-plain ids rejected) and only ever applied through the owner's per-tenant client — so an attacker-shaped id can at most touch a doc in the attacker's own dataset, never another tenant's project.
- **Defense in depth:** config reads go through the RLS-scoped client; the encrypted token is read only by the service-role client, keyed on the already-resolved `client_id`; the token is decrypted only after the user-scoped lookups succeed, and a tamper/wrong-key error propagates (never a plaintext fallback).
- **Fail closed, quietly.** Every resolver failure throws a typed `TenantResolutionError`; pages and actions turn those into friendly/generic states. The raw error, project id, token, and doc type are never rendered or returned.
- **GROQ injection guard.** `$type`/`$id`/`$draftId` are always bound parameters; all interpolated field names — read projections AND mutation object keys (incl. the slug container) — are validated by `assertSafeFieldPath` / `assertWritableFieldPaths` before use.
- **Data-loss guard.** Save does a read-modify-write overlay (non-essential client fields preserved); a body too rich for plain-text editing is detected and shown read-only so it is never silently stripped.
- **No secret values anywhere in the diff** (code, comments, tests, report) — grep-verified; no `'use client'` file imports a token/resolver/mutation/Sanity module — the build's `server-only` guard holds. The isolation/mutation tests use only invented, meaningless strings.
